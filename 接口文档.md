# 智能售后客服 Agent 接口文档

## 1. 文档信息

- 项目名称：智能售后客服 Agent
- 技术栈：Python、FastAPI、LangGraph、OpenAI/DeepSeek 兼容 LLM
- 默认服务地址：`http://127.0.0.1:8000`
- 鉴权方式：`Authorization: Bearer <token>`
- 数据格式：`application/json; charset=utf-8`

## 2. 通用约定

### 2.1 请求头

| Header | 必填 | 说明 | 示例 |
| --- | --- | --- | --- |
| `Authorization` | 是 | 用户 JWT Token，Agent 调用 Tool 时需要原样透传到业务接口 | `Bearer mock-user-token` |
| `Content-Type` | 有 JSON Body 时必填 | 请求体格式；`GET /api/order/status` 无请求体时不要求携带 | `application/json` |

### 2.2 通用响应结构

Agent 对外接口统一使用以下响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "request_id": "req_202606180001"
}
```

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `code` | integer | 业务状态码，`0` 表示成功，非 `0` 表示失败 |
| `message` | string | 状态说明 |
| `data` | object | 业务数据 |
| `request_id` | string | 请求追踪 ID |

### 2.3 通用错误码

| code | HTTP 状态码 | 说明 |
| --- | --- | --- |
| `0` | 200 | 成功 |
| `40001` | 400 | 请求参数错误 |
| `40101` | 401 | 未携带 Token 或 Token 无效 |
| `42201` | 422 | 缺少必要业务参数 |
| `50001` | 500 | Agent 未捕获系统异常；Tool 调用失败不得直接透出该错误 |
| `50301` | 200 | 下游业务系统繁忙，已由 Agent 兜底回复 |

> Agent 对话接口需要捕获 Tool 调用异常。除 FastAPI 入参校验和鉴权失败外，不应因为下游接口异常直接返回 `500`。

## 3. 对外 Agent 接口

### 3.1 非流式对话

- 接口：`POST /chat`
- 说明：接收用户自然语言输入，使用 LangGraph Router Node 将请求路由到闲聊/问答、订单查询、退款售后三个分支之一。
- 鉴权：必填 `Authorization: Bearer <token>`
- 正常或已兜底的 Tool 异常：HTTP `200`
- 未携带 Token 或 Token 无效：HTTP `401`

#### 请求参数

```json
{
  "session_id": "sess_001",
  "message": "帮我查一下订单 ORD-20260618-001 的物流状态",
  "stream": false
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `session_id` | string | 否 | 会话 ID，不传时服务端可自动生成 |
| `message` | string | 是 | 用户输入内容 |
| `stream` | boolean | 否 | 是否启用流式输出，非流式接口建议传 `false` |

#### 成功响应

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "session_id": "sess_001",
    "route": "order_status",
    "answer": "您的订单 ORD-20260618-001 当前状态为：已发货。",
    "entities": {
      "order_id": "ORD-20260618-001",
      "reason": null
    },
    "tool_calls": [
      {
        "name": "get_order_status",
        "status": "success",
        "latency_ms": 86
      }
    ],
    "error": null
  },
  "request_id": "req_202606180001"
}
```

#### 响应字段

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `data.session_id` | string | 会话 ID |
| `data.route` | string | 路由结果：`chat`、`order_status`、`refund` |
| `data.answer` | string | Agent 给用户的自然语言回复 |
| `data.entities.order_id` | string/null | 提取到的订单号 |
| `data.entities.reason` | string/null | 提取到的退款原因 |
| `data.tool_calls` | array | Tool 调用记录 |
| `data.error` | object/null | 已兜底处理的错误信息 |

#### 参数缺失响应示例

用户输入：`帮我退款，订单号是不知道`

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "session_id": "sess_001",
    "route": "refund",
    "answer": "请提供以 ORD- 开头的订单号，并说明退款原因，我再帮您提交退款申请。",
    "entities": {
      "order_id": null,
      "reason": null
    },
    "tool_calls": [],
    "error": {
      "type": "missing_required_entity",
      "detail": "order_id 或 reason 缺失"
    }
  },
  "request_id": "req_202606180002"
}
```

#### 幻觉参数拦截响应示例

用户输入：`帮我查订单 12345`

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "session_id": "sess_001",
    "route": "order_status",
    "answer": "您提供的订单号格式似乎不正确，正确的格式应为 ORD- 开头，请您核对后再试。",

```

#### 下游退款系统繁忙响应示例

```json
{
  "code": 50301,
  "message": "refund service unavailable",
  "data": {
    "session_id": "sess_001",
    "route": "refund",
    "answer": "抱歉，当前退款系统繁忙，请稍后再试。",
```

### 3.2 流式对话 SSE

- 接口：`POST /chat/stream`
- 说明：以 Server-Sent Events 返回 Agent 执行过程和最终回复；该接口对应需求文档中的可选加分项。
- 鉴权：必填 `Authorization: Bearer <token>`
- 响应类型：`text/event-stream`

#### 请求参数

```json
{
  "session_id": "sess_001",
  "message": "我要退款，订单 ORD-20260618-001，原因是商品破损"
}
```

#### 事件类型

| event | 说明 |
| --- | --- |
| `router` | 路由节点输出 |
| `entity` | 实体提取结果 |
| `tool_start` | Tool 开始调用 |
| `tool_end` | Tool 调用完成 |
| `message` | Agent 回复片段 |
| `error` | 已捕获并兜底处理的错误 |
| `done` | 流式输出结束 |

#### SSE 示例

```text
event: router
data: {"route":"refund"}

event: entity
data: {"order_id":"ORD-20260618-001","reason":"商品破损"}

event: tool_start
data: {"name":"apply_refund"}

event: message
data: {"delta":"您的退款申请已提交，"}

event: message
data: {"delta":"请等待平台审核。"}

event: done
data: {"request_id":"req_202606180005"}
```

## 4. Mock 业务接口

以下接口由本项目 FastAPI 本地模拟，作为 LangGraph Tool 的下游业务系统。

### 4.1 订单查询接口

- 接口：`GET /api/order/status`
- 说明：查询订单状态。
- 鉴权：必填 `Authorization: Bearer <token>`
- 成功：HTTP `200`
- 订单号格式错误：HTTP `400`
- 鉴权失败：HTTP `401`

#### Query 参数

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `order_id` | string | 是 | 订单号，必须以 `ORD-` 开头 |

#### 请求示例

```bash
curl -X GET "http://127.0.0.1:8000/api/order/status?order_id=ORD-20260618-001" \
  -H "Authorization: Bearer mock-user-token"
```

#### 成功响应（HTTP 200）

```json
{
  "order_id": "ORD-20260618-001",
  "status": "已发货"
}
```

#### 错误响应

订单号格式错误（HTTP 400）：

```json
{
  "detail": "order_id must start with ORD-"
}
```

未携带 Token 或 Token 无效（HTTP 401）：

```json
{
  "detail": "Unauthorized"
}
```

### 4.2 退款申请接口

- 接口：`POST /api/order/refund`
- 说明：提交退款申请，随机模拟成功或后端系统崩溃。
- 鉴权：必填 `Authorization: Bearer <token>`
- 成功：HTTP `200`
- 后端系统崩溃：HTTP `500`
- 鉴权失败：HTTP `401`

#### 请求参数

```json
{
  "order_id": "ORD-20260618-001",
  "reason": "商品破损"
}
```

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `order_id` | string | 是 | 订单号 |
| `reason` | string | 是 | 退款原因 |

#### 成功响应（HTTP 200）

```json
{
  "refund_id": "RF-20260618-001",
  "order_id": "ORD-20260618-001",
  "status": "submitted"
}
```

#### 后端崩溃响应（HTTP 500）

```json
{
  "detail": "refund backend crashed"
}
```

#### 未携带 Token 或 Token 无效（HTTP 401）

```json
{
  "detail": "Unauthorized"
}
```

## 5. LangGraph 状态设计

建议使用以下 State 字段管理对话历史、实体、路由、Tool 结果和错误状态：

```python
class AgentState(TypedDict, total=False):
    session_id: str
    user_message: str
    messages: list
    auth_token: str
    route: Literal["chat", "order_status", "refund"]
    order_id: str | None
    reason: str | None
    tool_result: dict | None
    error: dict | None
    answer: str
    request_id: str
```

## 6. 控制流设计

```text
START
  -> router_node
  -> conditional_edges:
       chat         -> chat_node         -> answer_node -> END
       order_status -> extract_node      -> validate_node -> order_tool_node  -> answer_node -> END
       refund       -> extract_node      -> validate_node -> refund_tool_node -> answer_node -> END
```

关键要求：

- `router_node` 只负责判断意图，不直接调用业务接口。
- `extract_node` 负责提取 `order_id`、`reason`。
- `validate_node` 使用 Pydantic 或等价逻辑做本地预校验，订单号不以 `ORD-` 开头时不发起 HTTP 请求。
- `order_tool_node` 和 `refund_tool_node` 必须从 State 读取 `auth_token`，调用 Mock 业务接口时写入 `Authorization` 请求头。
- Tool 捕获下游 `400`、`401`、`500` 等错误，写入 `state.error`，不向外抛出未处理异常。

## 7. Tool 异常处理规范

| 场景 | 处理方式 | 用户回复 |
| --- | --- | --- |
| 订单号缺失 | 不调用 Tool，写入 `missing_required_entity` | `请提供以 ORD- 开头的订单号，我再帮您查询。` |
| 订单号格式错误 | 本地预校验拦截，或捕获下游 `400` | `您提供的订单号格式似乎不正确，正确的格式应为 ORD- 开头，请您核对后再试。` |
| 未携带 Token | 对外接口返回 `401` | `Unauthorized` |
| 查询接口鉴权失败 | 捕获下游 `401` 并返回鉴权提示 | `登录状态已失效，请重新登录后再试。` |
| 退款接口返回 `500` | 捕获异常，后台打印 Error Trace，Agent 返回兜底话术 | `抱歉，当前退款系统繁忙，请稍后再试。` |
